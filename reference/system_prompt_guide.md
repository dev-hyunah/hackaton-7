# **Claude Code 기반 항공권 RM & 다이나믹 프라이싱 개발 가이드**

이 문서는 Anthropic의 CLI 개발 도구인 **Claude Code**를 활용하여 airline\_dynamic\_pricing\_rm\_guide.md에 기술된 AI 기반 RM 시스템을 실제로 구현하기 위한 단계별 개발 프롬프트 세트와 아키텍처 가이드입니다.

Claude Code는 스스로 파일 생성, 패키지 설치, 테스트 실행 및 디버깅을 수행할 수 있으므로, 아래 단계를 순서대로 프롬프트로 입력하며 점진적으로 개발을 진행하십시오.

## **0\. 권장 기술 스택 (Tech Stack)**

Claude Code에게 프로젝트를 맡기기 전에 사용할 기술 스택을 미리 정의해 둡니다.

* **Language**: Python 3.10+  
* **Framework**: FastAPI (실시간 GDS 동기화 및 API 서빙용)  
* **Math/Scientific**: NumPy, SciPy (EMSRb 역누적분포함수 계산용)  
* **Machine Learning / RL**: PyTorch, Gym (강화학습 최적화기 모델링용)  
* **Testing**: pytest

## **2\. 2단계: 일반석 EMSRb 인벤토리 제어 엔진 개발**

가이드북에 제시된 핵심 알고리즘인 EMSRb(Expected Marginal Seat Revenue \- b)를 수학적으로 정확히 구현하는 단계입니다.

### **💬 Claude Code 입력용 프롬프트 2**

\`app/core/emsr.py\` 파일을 생성하고 일반석 165석에 대한 EMSRb 알고리즘 엔진을 구현해 줘.

\[수학적 요구사항\]  
1\. 입력 데이터:  
   \- 클래스별 평균 요금: F1(정상 $Y$), F2(할인 $B$), F3(특가 $Q$)  
   \- 클래스별 잔여 수요의 평균과 표준편차: (mu1, sigma1), (mu2, sigma2), (mu3, sigma3)  
   \- 전체 공급 좌석 수 (기본값: 165\)  
2\. 계산 프로세스:  
   \- scipy.stats.norm의 역누적분포함수(ppf)를 사용해 줄 것.  
   \- Class 1 (정상 Y) 보호 수준 s1 계산:  
     s1 \= mu1 \+ sigma1 \* norm.ppf(1 \- F2/F1)  
   \- Class 1, 2 공동 보호 수준 s2 계산:  
     \- 합산 수요 mu\_12 \= mu1 \+ mu2, sigma\_12 \= sqrt(sigma1^2 \+ sigma2^2)  
     \- 가중 평균 요금 F\_bar\_12 \= (F1 \* mu1 \+ F2 \* mu2) / (mu1 \+ mu2)  
     \- s2 \= mu\_12 \+ sigma\_12 \* norm.ppf(1 \- F3 / F\_bar\_12)  
   \- 중첩 예약 제한량(Nested Booking Limits) 도출:  
     \- BL1 \= 165  
     \- BL2 \= max(0, 165 \- s1)  
     \- BL3 \= max(0, 165 \- s2)  
3\. 예외 처리:  
   \- 나눗셈 분모가 0이 되는 경우나 역누적분포함수 계산 시 범위를 벗어나는 오류에 대응하는 방어 코드를 작성해 줘.

구현 후, \`tests/test\_emsr.py\`를 작성하여 가이드의 임의 시나리오 값으로 테스트 코드를 작성하고 실행(pytest)해서 성공하는지 검증까지 완료해 줘.

## **3\. 3단계: WTP 및 다이나믹 프라이싱 모듈 개발**

프레스티지석(8석) 연속 가격 책정 및 일반석 가격 보정을 위한 Logit 기반 지불 용의(WTP) 모델을 설계합니다.

### **💬 Claude Code 입력용 프롬프트 3**

\`app/core/pricing.py\` 파일을 생성하고 프레스티지석의 "Pace-Based Continuous Pricing" 및 일반석의 "Logit 구매 확률 모델"을 개발해 줘.

\[세부 요구사항\]  
1\. Logit 구매 확률 함수 구현:  
   \- P(Buy | p) \= e^(alpha \- beta \* p) / (1 \+ e^(alpha \- beta \* p))  
   \- 가격 p, 수요 강도 alpha, 가격 민감도 beta를 입력받아 확률을 계산할 것.  
2\. 프레스티지석 가격 책정 함수 구현:  
   \- 공식: P\_prestige \= Base\_p \* K\_p \* (1 \+ theta \* (1 \- R\_p / 8))  
   \- 변수: Base\_p (기본가), K\_p (시간 흐름/수요에 따른 팩터), R\_p (잔여 좌석 수, 0\~8), theta (예약 속도 보정 계수)  
   \- 가이드라인 준수: 잔여 좌석(R\_p)이 4석 이하로 떨어지면 가격이 지수적으로 증가하도록 지수 가중치 팩터를 추가로 적용하는 로직을 결합해 줘.  
3\. 일반석 하이브리드 제어 가격 조정 함수 구현:  
   \- 특가(Q) 수요가 임계치를 넘을 때 Q를 무조건 차단하는 대신 가격을 B 클래스 하한선까지 강제로 올리는 상향 조정(Up-pricing) 논리 함수를 작성해 줘.

구현 후, \`tests/test\_pricing.py\`를 생성해 시뮬레이션 테스트를 수행하고 결과를 출력해 줘.

## **4\. 4단계: MDP 기반 강화학습(RL) 오케스트레이터 구현**

다양한 환경 상황 변화를 상태(State)로 인지하고 의사결정(Action)을 내려 보상(Reward)을 극대화하는 RL 프레임워크 스캐폴딩을 만듭니다.

### **💬 Claude Code 입력용 프롬프트 4**

\`app/models/rl\_optimizer.py\`에 RM을 위한 간단하고 경량화된 강화학습 환경(Gym Environment 스타일)과 최적화 오케스트레이터를 작성해 줘.

\[환경 세부 스펙\]  
1\. 상태(State): S\_t \= (t, R\_prestige, R\_1, R\_2, R\_3, competitor\_price)  
   \- t: 출발까지 남은 기간 (DTD)  
   \- R\_prestige: 프레스티지 잔여석  
   \- R\_1, R\_2, R\_3: 일반석 클래스별 잔여 예약 가능 수  
   \- competitor\_price: 경쟁사 동일 요금제 가격  
2\. 행동(Action):  
   \- 프레스티지석 가격 조정률 (-15% \~ \+80%)  
   \- 일반석 예약 제한량 벡터 조정 비율 \[BL1, BL2, BL3\]  
3\. 보상(Reward):  
   \- 해당 타입스텝에서 얻은 판매 매출액 (판매 수량 \* 가격)  
4\. AI 오케스트레이터:  
   \- 복잡한 외부 RL 라이브러리 의존 없이 구동 가능하도록 PyTorch를 활용한 매우 간단한 Q-Network(DQN) 혹은 휴리스틱 기반 폴리시 최적화 모듈을 구현해 줘.

동작성 확인을 위해 가상의 10단계 세션을 돌려보고 에피소드 누적 보상이 정상적으로 계산되는지 테스트를 수행해 줘.

## **5\. 5단계: FastAPI를 이용한 실시간 GDS API 구현**

GDS(Global Distribution System) 및 예매 시스템과 연동할 실시간 API 라우터와 제어용 시나리오 시뮬레이터를 개발합니다.

### **💬 Claude Code 입력용 프롬프트 5**

이제 개발한 알고리즘들을 하나로 묶어 FastAPI 앱(\`app/main.py\` 및 \`app/api/\`)을 완성해 줘.

\[필수 API 엔드포인트\]  
1\. POST \`/api/rm/optimize\`:  
   \- 입력: 현재 예약 상황(DTD, 잔여 좌석, 경쟁사 가격, 검색 유입량)  
   \- 작동: AI 예측 엔진 작동 \-\> EMSRb 계산 \-\> 최적 가격 및 예약 제한량(BL) 산정  
   \- 출력: 프레스티지 권장가, 일반석 클래스별(Y, B, Q) 예약 한도 및 권장가  
2\. POST \`/api/rm/simulate-scenario\`:  
   \- 가이드북의 '5. 실무 적용을 위한 AI 의사결정 시나리오'에 나오는 3가지 시나리오(A: 비수기 저조, B: 이벤트 검색폭발, C: 경쟁사 가격 인하)를 선택해 모의 실행하는 테스트 엔드포인트.  
   \- 각 시나리오별로 가이드북이 명시한 대로 조치 결과가 도출되는지 검증하는 API 인터페이스.

모든 소스 코드를 연결하고, \`uvicorn\`을 가상 터미널에서 구동하여 엔드포인트가 정상 실행되는지 \`curl\` 혹은 API 클라이언트 호출 스크립트를 만들어 자율적으로 통합 테스트를 완료해 줘.

## **6\. 개발 도중 에러가 발생했을 때 대처 프롬프트**

Claude Code가 코드 개발 중 예외 사항에 부딪히거나 터미널 테스트에서 실패(Fail)할 경우, 다음과 같이 해결하도록 유도합니다.

### **💬 Claude Code 입력용 프롬프트 (오류 수정용)**

방금 테스트에서 에러(또는 실패)가 발생했어.  
터미널 로그와 관련 수식들을 다시 확인한 다음,  
1\. 발생한 오류의 원인을 상세히 분석하고  
2\. EMSRb의 부동 소수점 에러나 음수 좌석 값 예외 등 경계값 조건들을 보완하여 코드를 수정한 뒤,  
3\. 테스트를 재실행해서 녹색 불(Pass)이 들어오는 것을 보여줘.
